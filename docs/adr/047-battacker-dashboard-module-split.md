# ADR-047: Battacker Dashboardのモジュール分割

## ステータス
採用

## コンテキスト
Battacker拡張のダッシュボードは単一ファイルに状態管理・データ取得・描画ロジックが集中しており、
変更時の影響範囲が広くレビューコストが高い。

## 決定
ダッシュボードを `hooks`・`tabs`・`components` に分割し、
状態管理と描画責務を分離する。

## 理由
- 変更点の局所化によりレビューとリグレッション確認のコストを下げる
- 画面単位の責務が明確になり設計の意図が読み取りやすい
- 将来のタブ追加やUI差し替え時に影響範囲を限定できる

## リスクと対応
- リスク: ファイル数増加により探索コストが上がる
- 対応（軽減）: `App.tsx` を単一の入口にし、タブ/部品の命名と配置を統一する
